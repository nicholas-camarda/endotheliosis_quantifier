## Context

The current quantification workflow already prepares scored examples, ROI crops, frozen glomeruli encoder embeddings, ordinal comparator outputs, burden-index outputs, and combined review artifacts through `eq run-config --config configs/endotheliosis_quantification.yaml`. It now uses explicit `subject_id`, `sample_id`, and `image_id` identity fields and grouped burden outputs under `burden_model/`.

The archived morphology change established two important facts:

- deterministic slit-style image processing is useful for QC but failed biological feature-readiness;
- downstream candidate metrics can look plausible even when the visual feature layer is biologically wrong.

That means the next quantification pass should optimize predictive grade-equivalent burden from learned ROI representations while keeping claim boundaries strict. The target remains predictive: infer the image-level grade/burden index from ROI pixels and masks. It is not a causal claim, a true pixel-area percent, or a validated mechanistic count of collapsed capillary lumina.

## Goals / Non-Goals

**Goals:**

- Add a learned ROI feature layer that extracts auditable feature tables from the existing ROI image/mask rows.
- Compare a capped phase-1 learned feature set using subject-heldout validation and grouped uncertainty calibration.
- Preserve all 707 admitted scored mask-paired rows while validating by `subject_id`.
- Separate per-image prediction readiness from subject/cohort burden-summary readiness.
- Generate review artifacts that make learned model behavior inspectable without pretending attribution proves mechanism.
- Keep deterministic morphology features available as QC/evidence while blocking their promotion when visual readiness fails.

**Non-Goals:**

- Do not create a new user-facing CLI.
- Do not claim true percent tissue involvement or a mechanistic closed-lumen count.
- Do not add compatibility aliases for old flat burden output paths.
- Do not require network downloads during the first implementation.
- Do not implement a full supervised capillary-lumen segmentation annotation workflow in this change.
- Do not fit optional torchvision, `timm`, DINO/ViT, UNI, CONCH, or other foundation/backbone candidates in phase 1. Audit them only unless a later explicit OpenSpec decision promotes one.

## Decisions

### Decision: Use learned ROI representations as the primary candidate family

The first implementation SHALL add `src/eq/quantification/learned_roi.py` to produce learned feature tables from ROI crops. It SHALL fit only the capped phase-1 candidate set:

- `current_glomeruli_encoder`: existing frozen encoder columns from `embeddings/roi_embeddings.csv`.
- `simple_roi_qc`: low-dimensional non-mechanistic ROI/mask/stain/quality features.
- `current_glomeruli_encoder_plus_simple_roi_qc`: the explicit hybrid of those two feature families.

The first implementation SHALL audit availability of additional local providers, but those providers SHALL remain audit-only and SHALL NOT create fitted candidates in this change.

Alternatives considered:

- Continue hand-engineering slit features. Rejected as the primary path because visual review showed boundary, nuclear, mesangial, and ubiquity failures.
- Train a new deep model end-to-end immediately. Deferred because the cohort has many image rows but only about 60 independent subject groups, so candidate screening and calibration need to be established before heavier training.
- Build a full pixel-level closed-lumen annotation workflow first. Deferred; this is appropriate if mechanistic closed-lumen quantification becomes the target, but it is not necessary to improve predictive grade-equivalent burden.
- Fit every available pretrained provider immediately. Rejected for phase 1 because about 60 independent subjects makes broad provider shopping a winner's-curse risk; provider audits are still useful, but promotion requires a separate explicit decision.

### Decision: Keep the workflow front door unchanged

The supported entrypoints remain:

- `eq run-config --config configs/endotheliosis_quantification.yaml`
- `eq quant-endo`

The new learned ROI branch SHALL be integrated into the existing quantification workflow and written under `burden_model/learned_roi/`.

### Decision: Audit providers before fitting candidates

`src/eq/quantification/learned_roi.py` SHALL write `burden_model/learned_roi/diagnostics/provider_audit.json`. The audit SHALL record whether each provider is available, skipped, failed, or audit-only, including import errors and model provenance. Candidate fitting SHALL only use phase-1 providers marked `available` and `fit_allowed`.

Initial provider IDs:

- `current_glomeruli_encoder`: current ROI embeddings generated by the workflow.
- `simple_roi_qc`: low-dimensional non-mechanistic ROI/mask/stain/quality features.
- `torchvision_resnet18_imagenet`: audit-only local torchvision ImageNet feature extractor if available.
- `timm_dino_or_vit`: audit-only local `timm` provider if installed and usable without downloads.

Pathology foundation models such as UNI/CONCH are valuable candidates, but the first implementation SHALL not require network downloads or unverifiable local weights. If local weights are absent, the provider audit SHALL record `unavailable`. If local weights are present, the provider audit SHALL record `available_audit_only`, not fit them.

### Decision: Use subject-heldout validation for every candidate

All image-level learned ROI candidates SHALL validate with `subject_id` grouped folds. Subject/cohort candidates SHALL aggregate one row per `subject_id` and validate held-out subjects. The report SHALL state row count and independent subject count separately.

### Decision: Readiness gates are track-specific

The learned ROI summary SHALL not select a candidate because it has the lowest error alone. It must record:

- stage-index MAE and grade-scale MAE;
- prediction-set coverage and average set size;
- numerical stability;
- cohort-stratified behavior;
- subject/cohort stability;
- whether per-image or subject/cohort track is selected.

Per-image readiness and subject/cohort readiness SHALL be independent verdicts.

The phase-1 image track SHALL be README/docs-ready only when all of the following pass:

- empirical prediction-set coverage `>= 0.88` overall at nominal 0.90 coverage;
- empirical prediction-set coverage `>= 0.80` for each observed-score stratum with at least 30 rows;
- empirical prediction-set coverage `>= 0.80` for each cohort with at least 30 rows;
- average prediction-set size `<= 4.0`, compared with the current baseline average set size `5.308`;
- stage-index MAE and grade-scale MAE both reported, with no selection based solely on stage-index recoding;
- no nonfinite predictions, no nonfinite feature columns used by the selected candidate, and no unresolved numerical-instability warnings;
- no cohort-confounding blocker from the diagnostics described below.

The phase-1 subject/cohort track SHALL be README/docs-ready only when all of the following pass:

- validation uses one aggregated row per `subject_id`;
- grouped bootstrap confidence intervals are written for subject/cohort summaries;
- no nonfinite predictions and no unresolved numerical-instability warnings;
- cohort-stratified residual and interval behavior does not trigger a cohort-confounding blocker;
- the report explicitly states that subject/cohort readiness does not imply precise operational per-image predictions.

### Decision: Add explicit cohort-confounding diagnostics

The learned ROI branch SHALL write `burden_model/learned_roi/diagnostics/cohort_confounding_diagnostics.json`. The diagnostics SHALL include:

- cohort counts by rows and `subject_id`;
- score distribution by cohort;
- candidate residual summaries by cohort;
- prediction-set coverage and average set size by cohort for image candidates;
- a simple cohort-predictability screen from the selected feature set;
- a leave-one-cohort-out or train-one-cohort/test-other diagnostic where each side has enough score support to estimate it without pretending the result is definitive.

A candidate SHALL be blocked from README/docs readiness when cohort diagnostics show clear cohort-specific failure. Clear failure means any of:

- cohort-specific prediction-set coverage `< 0.80` for a cohort with at least 30 rows;
- one cohort's grade-scale MAE exceeds the other cohort's grade-scale MAE by `>= 0.35`;
- selected features predict cohort with cross-validated balanced accuracy `>= 0.80` while score prediction remains weak or unstable;
- leave-one-cohort-out results fail to produce finite predictions or show qualitative collapse recorded in the diagnostic artifact.

### Decision: Evidence artifacts explain behavior without mechanistic overclaiming

`src/eq/quantification/learned_roi_review.py` SHALL write review artifacts under `burden_model/learned_roi/evidence/`. Required evidence includes nearest neighbors, high-error examples, high-uncertainty examples, cohort-stratified examples, and optional saliency/attention maps if the provider supports them. These artifacts SHALL be labeled as model-support evidence, not proof of endotheliosis mechanism.

### Decision: Deterministic morphology features stay blocked unless visual readiness passes

The learned ROI workflow may include deterministic morphology features as QC covariates only if their readiness status is carried forward. A blocked morphology feature family SHALL not be treated as biological closed-lumen evidence or used to justify a mechanistic claim.

### Decision: Iterate to verdict without expanding scope

Implementation SHALL iterate within the fixed phase-1 scope until the learned ROI workflow either:

- satisfies the explicit readiness gates defined in this change; or
- produces complete failure evidence showing which gates remain unmet.

Allowed iteration includes correcting implementation defects, improving calibration or reporting within the approved candidate set, and fixing artifacts required by the spec. Iteration SHALL NOT expand the fitted provider set, weaken validation gates, add compatibility paths, introduce new biological claims, or tune against held-out results without a new explicit OpenSpec decision.

## Risks / Trade-offs

- [Risk] Learned embeddings may improve predictive performance while remaining hard to interpret. -> Mitigation: require nearest-neighbor examples, high-error review, uncertainty summaries, and claim-boundary text.
- [Risk] Foundation-model providers may be unavailable in `eq-mac` or require network/model downloads. -> Mitigation: provider audit records availability; unavailable providers are not silently substituted.
- [Risk] Only 60 subject groups limits validation precision. -> Mitigation: subject-heldout validation is mandatory; reports separate row count from independent subject count.
- [Risk] Broad candidate search over optional providers can overfit the validation set. -> Mitigation: phase 1 fits only current encoder, simple ROI QC, and their hybrid; optional providers are audit-only.
- [Risk] Learned features may encode cohort, stain, acquisition, or mask-source differences. -> Mitigation: cohort-confounding diagnostics are required and can block readiness.
- [Risk] Per-image uncertainty may remain broad. -> Mitigation: candidate summaries must report average prediction-set size and coverage against the current baseline.
- [Risk] Subject-level summaries may look good while per-image predictions remain weak. -> Mitigation: track-specific readiness gates prohibit collapsing the two claims.
- [Risk] Saliency maps can be overinterpreted. -> Mitigation: label visual attribution as heuristic and never as causal/mechanistic proof.

## Migration Plan

1. Add learned ROI modules and tests without removing deterministic morphology modules.
2. Add provider audit and learned feature extraction under `burden_model/learned_roi/feature_sets/`.
3. Add subject-heldout learned candidate screens under `burden_model/learned_roi/candidates/` for the capped phase-1 candidate set only.
4. Add calibration and evidence artifacts under `burden_model/learned_roi/calibration/` and `burden_model/learned_roi/evidence/`.
5. Add cohort-confounding diagnostics under `burden_model/learned_roi/diagnostics/cohort_confounding_diagnostics.json`.
6. Update combined quantification review and README snippet gating to include learned ROI verdicts.
7. Rerun the full YAML workflow and record results in `audit-results.md`.

Rollback is not a compatibility branch. If learned ROI candidates fail readiness gates, keep the provider audit and candidate evidence, mark the learned ROI path exploratory, and do not promote a README/docs-ready quantification claim.

## Explicit Decisions

- New modules: `src/eq/quantification/learned_roi.py` and `src/eq/quantification/learned_roi_review.py`.
- New output root: `burden_model/learned_roi/`.
- New primary summary artifact: `burden_model/learned_roi/candidates/learned_roi_candidate_summary.json`.
- New provider audit artifact: `burden_model/learned_roi/diagnostics/provider_audit.json`.
- Existing workflow config remains `configs/endotheliosis_quantification.yaml`.
- The first implementation does not require network downloads.
- Deterministic morphology features remain QC/evidence unless visual feature-readiness gates pass.
- Phase-1 fitted candidate IDs are fixed as `image_current_glomeruli_encoder`, `image_simple_roi_qc`, `image_current_glomeruli_encoder_plus_simple_roi_qc`, `subject_current_glomeruli_encoder`, `subject_simple_roi_qc`, and `subject_current_glomeruli_encoder_plus_simple_roi_qc`.
- Optional provider audit statuses are fixed as `available_fit_allowed`, `available_audit_only`, `unavailable`, or `failed`.
- The current baseline for comparison is average prediction-set size `5.308` and overall coverage `0.898`; learned ROI image readiness requires average prediction-set size `<= 4.0` without coverage collapse.
